## Complexity Estimate

**M (5 points)** — Mudança transversal em 17 arquivos, com migração SQL e script de dados, mas padrão repetitivo.

## Risks

- **R1:** Tokens existentes ficam ilegíveis se a chave for perdida — mitigado com backward compat (decrypt detecta plaintext)
- **R2:** Performance overhead da encriptação — negligível (AES-256-GCM é hardware-accelerated via AES-NI)
- **R3:** Migração JSONB → TEXT no CRM pode quebrar queries — mitigado pois nenhuma query usa operadores JSONB nessa coluna

## Story

**As a** platform operator,
**I want** all OAuth tokens and API keys encrypted at rest in the database,
**so that** if the database is compromised, the tokens cannot be used to access external services.

## Scope

**IN:**
- Módulo de encriptação AES-256-GCM (`src/lib/security/encryption.ts`)
- Encriptar tokens no write: Gmail, Calendar, CRM, WhatsApp, API4COM
- Decriptar tokens no read: email.service, calendar.service, crm-sync.service, whatsapp.service, api4com.service, check-email-replies, fetch-gmail-signature
- Migração SQL: `crm_connections.credentials_encrypted` JSONB → TEXT
- Env var `TOKEN_ENCRYPTION_KEY` (opcional, graceful degradation)
- Script one-time para encriptar tokens existentes
- Testes unitários do módulo de encriptação

**OUT:**
- Rotação de chaves (futuro)
- Encriptação de outros campos sensíveis (e.g., webhook secrets)
- Column-level encryption no PostgreSQL (encriptação é no app layer)

## Acceptance Criteria

1. Novo módulo `encryption.ts` com `encrypt()`, `decrypt()`, `encryptJson()`, `decryptJson()` usando AES-256-GCM
2. Tokens escritos no banco estão no formato `<iv_hex>:<authTag_hex>:<ciphertext_hex>`
3. `decrypt()` detecta valores plaintext e retorna sem modificar (backward compatibility)
4. Sem `TOKEN_ENCRYPTION_KEY` configurada, sistema funciona normalmente (graceful degradation)
5. Todos os write paths de tokens usam `encrypt()` antes de salvar
6. Todos os read paths de tokens usam `decrypt()` antes de usar
7. `crm_connections.credentials_encrypted` convertida de JSONB para TEXT
8. Script `scripts/encrypt-existing-tokens.ts` encripta tokens existentes com `--dry-run` disponível
9. `pnpm typecheck` e `pnpm test:run` passam sem erros novos
10. Email de teste enviado com sucesso usando token encriptado+decriptado em produção

## Dev Notes

**Formato encriptado:** `<iv_hex(24)>:<authTag_hex(32)>:<ciphertext_hex(N)>` — cabe em coluna TEXT.

**Backward compatibility:** `decrypt()` verifica se o valor tem o formato `iv:tag:ciphertext` com tamanhos corretos. Se não, retorna o valor como-está. Isso permite deploy antes de rodar o script de migração — tokens plaintext continuam funcionando.

**Graceful degradation:** Se `TOKEN_ENCRYPTION_KEY` não estiver configurada, `encrypt()` retorna o valor sem modificar e `decrypt()` também. Dev local funciona sem configuração adicional.

**Deploy sequence:**
1. Deploy código (backward compatible)
2. Rodar migração SQL
3. Setar `TOKEN_ENCRYPTION_KEY` no Vercel
4. Redeploy
5. Rodar `scripts/encrypt-existing-tokens.ts`
