# Story 4.5: Upgrade & Downgrade Flows

## Status

**Ready for Review**

## Executor Assignment

executor: "@dev"
quality_gate: "@architect"
quality_gate_tools: ["lint", "typecheck", "test", "build"]

## Complexity Estimate

**M (5 points)** — Modais de confirmação com diff de features, lógica de verificação de uso atual, reativação.

## Risks

- **R1:** Downgrade pode desconectar integrações ativas — mitigado com aviso claro antes de confirmar
- **R2:** Stripe proration pode causar confusão de preço — mitigado mostrando valor pro-rated no modal

## Dependencies

- **Story 4.2** — Feature gating deve existir para o downgrade poder mostrar features que serão perdidas

## Story

**As a** manager,
**I want** clear upgrade and downgrade flows with feature impact warnings,
**so that** I can confidently change my plan knowing exactly what I'll gain or lose.

## Scope

**IN:**
- Modal de confirmação de upgrade com features que serão desbloqueadas
- Modal de confirmação de downgrade com features que serão perdidas
- Verificação de uso atual vs limites do novo plano no downgrade
- Página de reativação para subscriptions canceladas
- Toast de sucesso após mudança de plano (via query params do Stripe redirect)

**OUT:**
- Proration calculation (Stripe faz automaticamente)
- Coupon/discount codes
- Annual billing
- Cancellation recovery / win-back emails

## Acceptance Criteria

1. Botão "Fazer upgrade" abre modal mostrando features que serão desbloqueadas (diff entre plano atual e novo)
2. Botão "Mudar plano" para downgrade abre modal mostrando features que serão perdidas
3. Modal de downgrade verifica uso atual: se leads > novo max_leads, mostra aviso "Você tem X leads, o plano Y permite apenas Z"
4. Modal de downgrade verifica integrações: se CRM/Calendar conectado e novo plano não suporta, mostra aviso
5. Confirmar upgrade redireciona para Stripe Checkout com novo plano
6. Confirmar downgrade redireciona para Stripe Checkout com novo plano
7. Página `/upgrade` funciona para reativação de subscriptions canceladas (mostra 3 planos)
8. Toast de sucesso após retorno do Stripe Checkout (`?success=true`)
9. Toast de cancelamento após retorno sem pagar (`?canceled=true`)
10. `pnpm typecheck`, `pnpm lint` e `pnpm test:run` passam sem erros novos

## Dev Notes

**Plan diff structure:**
```typescript
interface PlanDiff {
  gained: Array<{ name: string; description: string }>;
  lost: Array<{ name: string; description: string }>;
  limitsChanged: Array<{ name: string; from: number | string; to: number | string }>;
}
```

**Exemplos de diff:**
- Starter → Pro: gained = ["CRM Integration", "Calendar Integration", "Enrichment Lemit"], limitsChanged = [leads: 1000→5000, AI: 50→200, WA: 500→2500]
- Pro → Starter: lost = ["CRM Integration", "Calendar Integration"], limitsChanged = [leads: 5000→1000, AI: 200→50, WA: 2500→500]

**Verificação de incompatibilidade:**
```typescript
const warnings: string[] = [];
if (currentLeadCount > targetPlan.max_leads) {
  warnings.push(`Você tem ${currentLeadCount} leads, mas o plano ${targetPlan.name} permite apenas ${targetPlan.max_leads}.`);
}
```
