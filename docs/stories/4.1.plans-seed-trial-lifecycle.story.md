## Complexity Estimate

**M (5 points)** — Migration de seed, reintrodução de `trialing`, cron de expiração, banner de trial, página de trial expirado.

## Risks

- **R1:** Reintroduzir `trialing` pode conflitar com subscriptions existentes que já são `active` — mitigado: migration apenas afeta novos signups
- **R2:** Trial sem cartão pode atrair spam/abuso — mitigado: trial de 14 dias com limites reduzidos do Starter
- **R3:** Cron de expiração pode afetar clientes pagantes — mitigado: WHERE clause filtra apenas `status = 'trialing'`

## Dependencies

- Nenhuma — esta é a story fundação do Epic 4

## Story

**As a** new user,
**I want** a 14-day free trial of the platform with clear limits and countdown,
**so that** I can evaluate the product before committing to a paid plan.

**As a** manager with an expiring trial,
**I want** clear warnings and an easy upgrade path,
**so that** I don't lose access to my data unexpectedly.

## Modelo de Cobrança (referência Meetime / Exact Sales)

| | Starter | Pro | Enterprise |
|--|---------|-----|------------|
| Preço base/mês | R$ 149 | R$ 349 | R$ 699 |
| Usuários inclusos | 4 | 4 | 4 |
| Usuário adicional/mês | R$ 49 | R$ 89 | R$ 129 |
| Max leads | 1.000 | 5.000 | 10.000 |
| AI/dia | 50 | 200 | Ilimitado |
| WhatsApp/mês | 500 | 2.500 | 10.000 |
| Enrichment | Básico | Intermediário | Completo |
| CRM | — | HubSpot/Pipedrive/RD | HubSpot/Pipedrive/RD |
| Calendar | — | Google Calendar | Google Calendar |

**Trial:** 14 dias gratuitos no plano Starter (sem cartão). Após expirar, acesso bloqueado até fazer upgrade.

## Scope

**IN:**
- Migration seed: garantir que 3 planos existem (idempotente, ON CONFLICT DO UPDATE para atualizar preços se mudaram)
- Reintroduzir status `trialing` no TypeScript (`SubscriptionStatus`)
- `handle_new_user()` volta a criar subscription como `trialing` com `current_period_end = now() + 14 days`
- Cron job pg_cron diário: `trialing` → `canceled` quando `current_period_end < now()`
- Banner de trial no app header: "Seu trial expira em X dias — Fazer upgrade" (apenas `trialing`)
- Página `/upgrade` para trial expirado (reutilizar `PlanComparison` existente)
- `SubscriptionGuard` redireciona `trialing` expirado para `/upgrade`
- Notificação in-app quando faltam 3 dias para expirar

**OUT:**
- Grace period pós-trial
- Email de aviso de trial expirando (futuro)
- Planos customizados / enterprise sob consulta
- Cobrança anual (desconto por anuidade)
- Implementação cobrada (onboarding pago)

## Acceptance Criteria

1. Migration insere/atualiza 3 planos com limites corretos (idempotente via ON CONFLICT)
2. Novo signup cria subscription `trialing` com `current_period_end = now() + 14 days`
3. `SubscriptionStatus` TypeScript inclui `'trialing'`
4. Cron executa diariamente e muda trials expirados para `canceled`
5. Banner amarelo de trial visível no app header com countdown ("Trial: X dias restantes")
6. Banner não aparece para subscriptions `active` (pagantes)
7. Quando trial expira (`canceled` via cron), `SubscriptionGuard` redireciona para `/upgrade`
8. Página `/upgrade` mostra 3 planos com CTA de checkout
9. Notificação in-app criada quando faltam 3 dias para trial expirar
10. `pnpm typecheck`, `pnpm lint`, `pnpm test:run` e `pnpm build` passam

## Dev Notes

**Status enum no PostgreSQL:**
O enum `subscription_status` no banco JÁ contém `'trialing'` — ele nunca foi removido (PostgreSQL não suporta DROP VALUE em enums). A migration `20260303120000` apenas fez UPDATE dos registros existentes de `trialing` para `active`.

**Modelo de cobrança (Meetime/Exact Sales style):**
- Preço base inclui 4 usuários (seats)
- Usuários adicionais cobrados por cabeça/mês
- Limites por volume (leads, AI, WhatsApp) por plano
- Trial 14 dias sem cartão
- Após trial: bloqueio total até checkout

**Trial flow:**
```
Signup → trialing (14d) → [paga?] → active
                        → [não paga?] → cron cancels → /upgrade
```

**Banner calculation:**
```typescript
const daysRemaining = Math.max(0, Math.ceil(
  (new Date(periodEnd).getTime() - Date.now()) / (1000 * 60 * 60 * 24)
));
```

**Cron endpoint pattern (reutilizar padrão existente):**

  return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
