## Story
**As a** gestor (manager),
**I want** parear um número WhatsApp dedicado a cada SDR via QR code,
**so that** cada SDR tenha identidade própria de chamada e o risco de ban fique distribuído.

## Complexity
**M — Medium** (proxy server-side + QR/polling + persistência de sessão)

## Scope
**IN:**
- Server Action/route proxy que injeta a `WACALLS_API_KEY` (browser nunca fala direto com o microserviço).
- Criar sessão (`POST /api/sessions`), exibir **QR**, pollar status (`GET /api/sessions`) até `paired`.
- Persistir/atualizar `whatsapp_call_sessions` (status `disconnected|pairing|connected`).
- Re-parear sessão morta (`POST /api/sessions/{sid}/pair`).
- Tela de gestão (manager-only via `requireManager()`).

**OUT:**
- Pool multi-número.
- Auto-recuperação sem intervenção (monitor fica em 7.9).

## Dependencies
- 7.1 (microserviço com `/api/sessions*` + API key).
- 7.2 (tabela `whatsapp_call_sessions`).

## Risks
| Risco | Prob. | Impacto | Mitigação |
|-------|-------|---------|-----------|
| API key vazar pro browser | Média | Crítico | todo acesso ao serviço via proxy server-side; key só em env do servidor |
| QR expira antes do scan | Média | Baixo | polling + botão "gerar novo QR" |
| Sessão cai e SDR não percebe | Média | Médio | status persistido + badge; monitor em 7.9 |

## Acceptance Criteria
1. Manager acessa tela de "Números WhatsApp" (manager-only); SDR comum não acessa.
2. "Parear número" chama o proxy → `POST /api/sessions` → exibe QR retornado pelo serviço.
3. O sistema polla status e, ao `paired`, persiste `whatsapp_call_sessions` (`status='connected'`, `phone_number`, `service_session_id`, `paired_at`).
4. O browser **nunca** recebe a API key; toda chamada ao microserviço passa por route/action server-side.
5. Re-parear uma sessão `disconnected` gera novo QR sem criar registro duplicado.
6. Cada SDR tem no máximo 1 sessão ativa (constraint/validação).
7. RLS: SDR só vê a própria sessão; manager vê todas da org.
