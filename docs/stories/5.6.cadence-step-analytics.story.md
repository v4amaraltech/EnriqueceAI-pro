## Complexity Estimate

**M (5 points)** — Novo serviço de agregação por step, componente de tabela detalhada, integração com filtros existentes, queries com GROUP BY em interactions.

## Risks

- **R1:** Queries agrupando interactions por step_id em períodos grandes podem ser lentas — mitigado: índice `idx_interactions_step_variant(step_id, type)` já existe
- **R2:** Cadências com muitos steps (10+) podem gerar tabelas extensas — mitigado: expandir/colapsar por cadência, scroll horizontal se necessário
- **R3:** Steps sem interactions no período podem não aparecer — mitigado: LEFT JOIN com cadence_steps para mostrar todos os steps mesmo com zero dados

## Dependencies

- **Story 5.1** — DateRangePicker (filtro de período)
- **Story 5.5** — AnalyticsFilters (filtros de SDR e cadência)

## Story

**As a** manager,
**I want** to see detailed performance metrics for each step of my cadences,
**so that** I can identify which steps drive engagement and optimize my outreach sequences.

## Scope

**IN:**
- Tabela detalhada por step dentro de cada cadência com métricas: Enviados, Abertos, Clicados, Respondidos, Reuniões, Taxa de Abertura %, Taxa de Resposta %
- KPI cards no topo: Total Enviados, Total Engajados (abriu/clicou/respondeu), Taxa de Engajamento %
- Drill-down: clicar em uma cadência na `CadencePerformanceTable` existente expande os steps
- Filtros existentes do `AnalyticsFilters` (período, SDR, cadência) aplicados
- Server action `fetchStepLevelAnalytics` com agregação por `step_id` + `type`
- Integração na página `/statistics/prospecting/cadences`

**OUT:**
- Métricas de A/B testing por step (story futura)
- Gráfico de funil visual por steps (mantém tabela por simplicidade)
- Estatísticas de WhatsApp engagement (WhatsApp não tem open/click tracking confiável)
- Comparação período-a-período nos steps (pode usar DeltaIndicator da 5.2 futuramente)

## Acceptance Criteria

1. Ao clicar em uma cadência na tabela existente, expande mostrando métricas por step
2. Cada step mostra: Passo #, Atividade (canal + nome), Enviados, Abertos, Clicados, Respondidos, Reuniões
3. Taxas calculadas: Abertura % (abertos/enviados), Click % (clicados/enviados), Resposta % (respondidos/enviados)
4. KPI cards no topo da página: Total Enviados, Engajados (leads únicos com open/click/reply), Taxa de Engajamento %
5. Filtros de período, SDR e cadência funcionam corretamente com os dados por step
6. Steps sem dados no período aparecem com zeros (não são omitidos)
7. Loading skeleton durante carregamento dos dados de step
8. Testes unitários para o serviço de agregação e componente de tabela
9. `pnpm typecheck`, `pnpm lint`, `pnpm test:run` e `pnpm build` passam

## Dev Notes

**Índice existente (performance garantida):**
```sql
CREATE INDEX idx_interactions_step_variant ON interactions (step_id, type) WHERE step_id IS NOT NULL;
```

**Query base para step analytics:**
```typescript
const { data } = await supabase
  .from('interactions')
  .select('step_id, type')
  .eq('org_id', orgId)
  .eq('cadence_id', cadenceId)
  .gte('created_at', from)
  .lte('created_at', to)
  .not('step_id', 'is', null);

// Agrupar client-side por step_id + type (Supabase JS não suporta GROUP BY)
```

**Steps data (para labels):**
```typescript
const { data: steps } = await supabase
  .from('cadence_steps')
  .select('id, step_order, channel, activity_name')
  .eq('cadence_id', cadenceId)
  .order('step_order', { ascending: true });
```

**Conceito de "Engajado" (Meetime):**
Lead que teve qualquer interação positiva no step: `type IN ('opened', 'clicked', 'replied', 'meeting_scheduled')`. Contar leads **únicos** (DISTINCT lead_id).
