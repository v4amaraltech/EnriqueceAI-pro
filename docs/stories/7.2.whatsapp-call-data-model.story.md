## Story
**As a** plataforma EnriqueceAI,
**I want** um discriminador de provider no passo de ligação e uma tabela de sessões WhatsApp por SDR,
**so that** o sistema saiba quando renderizar o discador WhatsApp e qual número/sessão usar por SDR.

## Complexity
**S — Small** (2 migrations forward-only)

## Scope
**IN:**
- `cadence_steps.call_provider text NULL` (NULL = ligação PSTN/API4COM; `'whatsapp'` = discador WhatsApp).
- Tabela `whatsapp_call_sessions` (1 número ↔ SDR ↔ sessão do microserviço) + RLS + trigger `set_updated_at`.
- Tipos TS correspondentes (`CadenceStepRow.call_provider`, novo tipo de sessão).

**OUT:**
- Mudança no enum `channel_type` (decisão: reusar `phone`).
- Tabela nova de call events (reusa `calls`).
- UI (fica em 7.3/7.4).

## Dependencies
- Nenhuma (pode rodar em paralelo com 7.1).

## Risks
| Risco | Prob. | Impacto | Mitigação |
|-------|-------|---------|-----------|
| Tipos TS desatualizados pós-migration | Média | Baixo | regenerar types ou documentar `TYPES_STALE` |
| RLS incorreta expondo sessão de outro SDR | Baixa | Alto | policy org-scoped via `public.user_org_id()`; SDR vê só a sua, manager vê todas |

## Acceptance Criteria
1. Migration adiciona `cadence_steps.call_provider text` (nullable, default NULL) de forma idempotente.
2. `CREATE TABLE whatsapp_call_sessions` com: `id, org_id (FK organizations ON DELETE CASCADE), user_id, service_session_id text, phone_number text, status text DEFAULT 'disconnected', paired_at timestamptz, created_at, updated_at`.
3. Tabela tem RLS habilitada; SELECT/ALL org-scoped (`public.user_org_id()`); manager (`public.is_manager()`) enxerga todas as sessões da org.
4. Trigger `set_updated_at BEFORE UPDATE ... EXECUTE FUNCTION update_updated_at()` aplicada.
5. Migration wrapped em `BEGIN/COMMIT`, timestamp 14 dígitos único, `IF NOT EXISTS`/`CREATE OR REPLACE`.
6. Tipos TS em `src/features/cadences/types` e novo módulo refletem as colunas; `pnpm typecheck` passa.

## Dev Notes
- Convenções canônicas: função `update_updated_at()`, trigger `set_updated_at`, RLS `public.user_org_id()`/`public.is_manager()`.
- **Desvio do plano (intencional):** `whatsapp_call_sessions.phone_number` é **NULLABLE** (no plano B.2 estava NOT NULL). Motivo: ao criar a sessão (`POST /api/sessions`) só temos `service_session_id`; o número (JID) só é conhecido após o pareamento QR. Documentado no comentário da coluna.
- `call_provider` é `text` com CHECK `(NULL | 'whatsapp')` em vez de novo enum — extensível e sem o gotcha de `ALTER TYPE` em transação.
