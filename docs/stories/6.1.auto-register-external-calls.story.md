## Story
**As a** gestor de vendas,
**I want** que ligações feitas pelo SDR fora da plataforma (direto no webphone API4COM) sejam automaticamente registradas como atividades no Enriquece AI,
**so that** eu tenha visibilidade completa da produtividade real do time e o histórico do lead fique completo.

## Complexity
**M — Medium** (7 tasks, webhook extension + phone matching + cadence integration + UI badge)

## Scope

**IN:**
- Processar ligações outgoing externas via webhook API4COM
- Matching de lead por telefone (3 fontes)
- Registro de ligação órfã quando lead não encontrado
- Avanço automático de cadência (step phone)
- Badge visual "Externa" na timeline
- Mapeamento ramal → user_id via migration

**OUT:**
- Ligações incoming (lead → SDR) — não processadas
- UI para gerenciar mapeamento de ramais (feito via migration/seed)
- UI para visualizar/gerenciar ligações órfãs (backlog futuro)
- Edição manual de ligações externas
- Notificação push para o SDR sobre ligação registrada

## Dependencies
- Webhook API4COM funcional e recebendo eventos `channel-hangup`
- Tabela `calls` e `interactions` existentes
- Mapeamento ramal → user_id (criado na Task 2 desta story)

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Falso positivo no matching (número compartilhado entre leads) | Média | Médio | Usar match exato nos últimos 8+ dígitos; se múltiplos leads, vincular ao mais recente (último `updated_at`) |
| Performance da busca em JSONB `socios[].celulares[]` | Baixa | Baixo | Busca ocorre no webhook (async), não impacta UI. Se necessário, criar GIN index |
| Ramal não mapeado no banco | Alta (novos SDRs) | Médio | Criar call órfã sem `performed_by`, logar warning para o gestor revisar |
| API4COM não envia `direction` no payload | Baixa | Alto | Inferir outgoing pelo `callerIdNumber` ser um ramal interno (4 dígitos) |

## Acceptance Criteria

1. Quando o webhook API4COM recebe um `channel-hangup` de uma ligação outgoing que **não** tem call registrada no Enriquece AI, o sistema busca o lead pelo número de destino
2. A busca de lead por telefone deve verificar: `leads.telefone`, `leads.phones[]` (JSONB), e `leads.socios[].celulares[]`
3. Se encontrar o lead, cria automaticamente uma `interaction` do tipo `sent` no canal `phone` com metadata `{ source: 'external_api4com' }`
4. Se **não** encontrar o lead, cria um registro de ligação "órfã" na tabela `calls` com `lead_id: null` para revisão do gestor
5. Apenas ligações **outgoing** (SDR → lead) são processadas. Incoming são ignoradas
6. Se o lead está em uma cadência ativa com próximo step = `phone`, a ligação externa **avança o step** automaticamente
7. Na timeline do lead, ligações externas exibem indicador visual "Via API4COM (externa)" para distinguir de ligações feitas pela plataforma
8. A normalização de telefone para matching usa a mesma lógica existente (`normalizePhone` — remove DDI, DDD, formatação)
9. O SDR é identificado pelo `ramal` do webhook, mapeado via `api4com_extension` no `organization_members` ou configuração equivalente

### Fluxo Atual do Webhook
1. API4COM envia `channel-hangup` → `route.ts` processa
2. Busca call existente por `api4com_call_id` em `calls`
3. Se encontra: atualiza com duração, status, recording
4. Se **não** encontra: atualmente ignora → **este é o ponto de extensão**
