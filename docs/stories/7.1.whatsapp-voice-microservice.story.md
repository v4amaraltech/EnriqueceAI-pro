## Story
**As a** plataforma EnriqueceAI,
**I want** um microserviço Go isolado que faça chamadas de voz 1:1 via WhatsApp (WebRTC ↔ MLow ↔ SRTP do WhatsApp),
**so that** o front do Enriquece consiga iniciar/encerrar ligações via REST sem acoplar whatsmeow/pion ao runtime Next.js.

## Complexity
**L — Large** (fork + 3 reimplementações de produção + deploy VPS com rede)

## Scope
**IN:**
- Fork do WaCalls (MIT) como repo separado proprietário.
- Auth por **API key** (`X-API-Key` header + `?apiKey=` p/ SSE) — reimplementado (referência AstraCalls, sem copiar código AGPL).
- **Webhook por sessão** (`GET/POST/DELETE /api/sessions/{sid}/webhook`) emitindo lifecycle de chamada — reimplementado.
- Fix **NAT 1:1 / ICE-TCP** (`WACALLS_PUBLIC_IP=auto`, `WACALLS_UDP_PORT`) p/ VPS que bloqueia UDP inbound.
- Deploy Docker (network=host) + HTTPS (Traefik/socat) na VPS.

**OUT:**
- Postgres por sessão estilo WAHA (SQLite do WaCalls basta no MVP de 1 número).
- API de mensagens, Chatwoot, widget.js do AstraCalls.
- Perna de IA (PCM 16kHz → Gemini/Pipecat).
- Multi-número/pool.

## Dependencies
- VPS provisionada com IP público e portas liberadas (UDP + TCP fallback).
- Decisão de licença travada (base MIT, sem importar AGPL).

## Risks
| Risco | Prob. | Impacto | Mitigação |
|-------|-------|---------|-----------|
| UDP inbound bloqueado quebra mídia WebRTC | Alta | Alto | ICE-TCP no mesmo port + `SetNAT1To1IPs` |
| Instabilidade de sessão whatsmeow | Média | Médio | endpoint de re-pareamento + SSE de status |
| Sem auth → exposição | Alta | Crítico | API key obrigatória em todas as rotas `/api/*` |
| Contaminação AGPL | Baixa | Alto | reimplementar auth/webhook/NAT, não copiar do AstraCalls |

## Acceptance Criteria
1. O serviço sobe via `docker stack deploy`/compose com `WACALLS_PUBLIC_IP`, `WACALLS_UDP_PORT`, `WACALLS_API_KEY` configurados.
2. Todas as rotas `/api/*` exigem API key válida; sem key → 401.
3. `POST /api/sessions` cria sessão e emite QR; `GET /api/sessions` reporta `paired`/status.
4. `POST /api/sessions/{sid}/calls {phone, record?}` inicia chamada e retorna `call_id`; `POST .../webrtc` troca SDP; `DELETE .../calls/{id}` encerra.
5. `GET /api/events` (SSE) emite eventos de lifecycle: `ringing`, `active/answered`, `ended`, `rejected`, `no-answer`.
6. Webhook por sessão dispara POST com o mesmo lifecycle pro destino configurado.
7. Com UDP inbound bloqueado na VPS, a chamada ainda conecta via ICE-TCP (validado num teste real).
8. Uma chamada de voz 1:1 real completa: SDR (browser) ↔ número WhatsApp de teste, áudio bidirecional.
